# tidy — 安全なディレクトリ整理

散らかったディレクトリを、**中身を読んで意味的に分類**し、安全に整理するスキル。

## 大原則（必ず守る）

このスキルの価値は「壊さないこと」と「全部追えること」にある。ユーザーは過去に
「どこいったか分からなくなる」整理で困っている。だから:

1. **削除しない** — 不要ファイルはゴミ箱ではなく `_捨て/` ディレクトリへ移すだけ。
   最終的に消すかどうかはユーザーが自分で判断する。
2. **実行前に必ず確認** — `preview` で計画を提示し、日本語で要約してユーザーの承認を得てから
   `apply` する。勝手に動かさない。
3. **全移動を記録** — `apply` は manifest を残す。「あのファイルどこ行った?」に常に答えられ、
   `undo` で完全に元へ戻せる。
4. **取り残しを作らない** — 対象内の全ファイルが分類対象に入っているか確認する。

スクリプトは `scripts/organize.py`（Python 標準ライブラリのみ・依存ゼロ）。

## ふたつのモード

依頼内容に応じて使い分ける。どちらも **scan → suggest → preview → apply → verify** の流れは同じ。
やり直したくなったら **undo / redo**。

- **その場整理**: 1つの散らかったフォルダを、その中で分類する。plan の `from` は相対パス。
- **集約 + 重複排除**: 講義資料などが**複数の場所に散らばって重複している**のを、1か所に集めて
  同一内容の重複を1つに絞る。plan の `from` は絶対パス、`to` は宛先ルートからの相対パス。
  → 詳細は下の「集約モード」を参照。

## ワークフロー（その場整理）

### 1. まず診断（assess）— skill向きか手動向きか
整理を始める前に、**必ず**対象ディレクトリを診断する。これは「触ると壊れる場所に skill を
かけてしまう」事故を防ぐための安全装置。
```
python3 scripts/organize.py assess "<対象dir>"
```
判定 `[verdict]` の意味と対応:
- **NG**: コードプロジェクト（`.git`/`package.json` 等）やシステム/ライブラリ配下。
  → **整理しない**。理由を伝え、「対象を具体的なサブフォルダに絞る」か「手動」を勧める。
  どうしても進める場合はユーザーに明確な確認を取る。
- **MANUAL**: ファイルが少ない。→ 自動化の利点が薄いので手動を勧める（やってと言われればやる）。
- **MAYBE**: どちらでも。→ 中身を見て提案し、ユーザーに選んでもらう。
- **SKILL**: 散らかっていて自動整理が有効。→ そのまま次へ進む。
- **SKIP**: 空。

診断結果と推奨をユーザーに伝えてから次に進む。深い階層まで見るか（`--depth`）もここで確認する。

### 2. スキャン
```
python3 scripts/organize.py scan "<対象dir>" --out /tmp/scan.json
```
複数の場所をまとめて見る場合はディレクトリを並べる（集約モード）:
```
python3 scripts/organize.py scan ~/Downloads ~/Desktop "~/Documents/講義" --out /tmp/scan.json
```
`scan.json` には各ファイルの `abspath`・`path`（所属rootからの相対）・`root`・サイズ・更新日時・
拡張子・バイナリ判定・**先頭スニペット**・`sha1`・ゴミ候補フラグ・`course_hint`（講義資料らしさ）が入る。
`duplicates` には**場所をまたいで**同一内容（同 sha1）のファイル組が絶対パスで並ぶ。

scan は **コード依存（import/参照）も解析**する。各コードファイル (`.py .js .ts .tsx .html .css .md .sh`)
を読み、resolve できた依存先を `imports` / `imported_by` として埋める。さらに連結したファイル群を
`code_dependencies.clusters` としてまとめる（例: `main.py` + `helper.py` + `utils.py` が1クラスタ）。
解析対象:
- Python: `from X import Y`（dot relative 含む）/ `import X`（sibling 慣習）
- JS/TS: `import ... from './X'` / `require('./X')` / `import('./X')`
- HTML: `src=` / `href=` の **ローカル参照**
- CSS: `@import` / `url(...)`
- Markdown: 画像 `![](./X.png)` の **ローカル参照**

クラスタ情報は次の `suggest` で「import が壊れないように一緒に配置する」ために使われる。
巨大ディレクトリでスキャンを速くしたい場合は `--no-deps`。

### 3. baseline を suggest で生成し、必要なら手で直す
まず `suggest` で機械的に baseline plan を作る。拡張子・ファイル名・スニペット・重複情報から
妥当な行き先を埋めてくれるので、ここから差分修正するだけで済む。**ゼロから plan を手書きしない**。
```
python3 scripts/organize.py suggest --in /tmp/scan.json --out /tmp/plan.json
```
集約モード（複数 root を scan した場合）は宛先ルートを `--target` で指定:
```
python3 scripts/organize.py suggest --in /tmp/scan.json --out /tmp/plan.json --target ~/Documents/講義資料
```
出力された `plan.json` を読み、必要に応じてエディットする。修正ポイント:
- **判定不能の `skipped` を救う**: suggest は迷ったら除外する。ユーザーに必要なものは plan に追記。
- **境界ケースの再判定**: 同じ拡張子でも中身を見れば行き先が違うこと（例: `.json` が設定か
  データか、`.md` が仕様書かメモか）。スニペットを根拠に書き換える。
- **`_捨て/` 行きの最終確認**: 重複の正本選定や junk 判定をユーザー視点で見直す。
- **クラスタを分断しない**: suggest はコード依存クラスタ（`main.py` + `helper.py` 等）を同じ
  subfolder にまとめて配置する。手で `to` を変えるときは**クラスタの全メンバーを同じ宛先に
  揃える**こと。バラすと import が壊れる。preview が分断を検出して警告するので、警告が
  出たら必ず修正する。
- **取り残しを最小化**: plan に入っていないファイルは元の場所に残る。「全部きれいにしたい」
  なら scan.json の全ファイルが plan に登場するようにする。

`plan.json` の形式（suggest が生成するもの・手書きも同じ）:
```json
{
  "root": "<対象dirの絶対パス>",
  "trash_dir": "_捨て",
  "moves": [
    {"from": "古い仕様メモ.md", "to": "ドキュメント/仕様/古い仕様メモ.md", "reason": "仕様書の下書き"},
    {"from": "scan_tmp.log",    "to": "_捨て/scan_tmp.log",               "reason": "一時ログ・不要"}
  ]
}
```
`from`/`to` は対象dir からの相対パス。各 move に短い `reason`（日本語）を付けると、後で
ユーザーが判断を追える。

分類カテゴリと「いる/いらない」の判定基準は `references/classification.md` を参照（境界ケースの
判断時に読む）。

### 4. プレビュー（dry-run）して確認を取る
```
python3 scripts/organize.py preview "<対象dir>" --in /tmp/plan.json
```
宛先別ツリー・件数・宛先衝突・**plan に含まれない取り残しファイル**を表示する（移動はしない）。
出力を読み、ユーザーに日本語で要約して提示する。例:

> 「`ドキュメント/` に 12 件、`画像/` に 8 件、`コード/` に 3 件を移動します。
>  `_捨て/` には `.DS_Store` など不要候補 5 件を隔離します（削除はしません）。
>  未分類で元の場所に残るファイルが 2 件あります。このまま進めてよいですか?」

承認が得られるまで `apply` しない。修正要望があれば plan.json を直して再 preview する。

### 5. 適用
```
python3 scripts/organize.py apply "<対象dir>" --in /tmp/plan.json --yes
```
移動を実行し、`<対象dir>/_整理ログ/manifest-<日時>.json` に全移動を記録する。
宛先が衝突した場合は上書きせず ` (1)` などを付けてリネームする。Ctrl-C 中断時は
それまでの移動を manifest に残してから抜けるので、`undo` で復元できる。

### 6. 検証
apply 後、念のため整合性を確認する。各 from が消えて各 to が存在することを機械的にチェック:
```
python3 scripts/organize.py verify "<対象dir>"
```
1件でも欠けがあれば rc=1 で詳細が出る。問題があれば `undo` してから plan を見直す。

### 7. 報告
- 何件をどこへ移したか、`_捨て/` に何件隔離したかを伝える（`verify` 結果を併記）。
- 「`_捨て/` の中身を確認して、不要なら自分で削除してください」と案内する。
- 「`undo` で全部元に戻せます」と必ず伝える:
  ```
  python3 scripts/organize.py undo "<対象dir>"
  ```
- undo を使ったあとで「やっぱり apply 後がよかった」となったら `redo` で apply 後の状態へ戻せる:
  ```
  python3 scripts/organize.py redo "<対象dir>"
  ```

### 8. _捨て の見回り（review）
`_捨て/` は時間とともに溜まる。`review` で中身を一覧し、必要なら復元 / 物理削除する:
```
python3 scripts/organize.py review "<対象dir>"
  # → 各ファイルの隔離日時・元の場所・理由が出る
python3 scripts/organize.py review "<対象dir>" --restore "*.pdf" --yes
  # → pattern 一致を元の場所へ戻す（apply と同じ衝突回避ロジック）
python3 scripts/organize.py review "<対象dir>" --purge --older-than 30 --yes
  # → 30日より古いものを物理削除（tidy の唯一の delete 経路。必ず --yes 必須）
```
`review` を案内するタイミング: apply 直後の「報告」の末尾、あるいは「最近 _捨て に何ある？」と聞かれたとき。
`--purge` は不可逆なので、ユーザの明示的依頼があったときだけ提案する。

## 集約モード（複数の場所から集めて重複排除）

「講義資料があちこちにあって重複してる、まとめて整理したい」といった依頼向け。

1. **集める場所と宛先を決める**。どのフォルダを探すか（例: `~/Downloads`, `~/Desktop`,
   `~/Documents`）をユーザーに確認し、**集約先ルート**（例: `~/Documents/講義資料`）を決める。
   宛先ルートは新規でも既存でもよい。`_捨て/` と `_整理ログ/` はこのルート配下に作られる。
2. **複数ディレクトリを scan**（上記の通りディレクトリを並べる）。
3. **plan.json を作る**。`root` を宛先ルートにし、各 move の `from` は**絶対パス**（scan の
   `abspath`）、`to` は宛先ルートからの相対パスにする。
   - **重複排除**: `duplicates` の各組で、正本を1つ選んで講義フォルダへ。**残りは `_捨て/` へ**
     （削除ではなく隔離）。正本は「分かりやすい名前」「新しい更新日時」「元から正しい場所にある」
     ものを優先。
   - **集約先の構造**: 科目ごとに分けると探しやすい。例:
     `線形代数/第3回.pdf`、`ミクロ経済/講義05.pdf`。`course_hint` が True のものを中心に、
     スニペットや名前で科目を判断する。
   - 講義と無関係なファイルは **plan に入れない**（元の場所に残る）。集約モードでは宛先ルートの
     外にあるファイルは取り残し警告の対象外なので、無関係ファイルを誤って動かす心配はない。

   ```json
   {
     "root": "/Users/you/Documents/講義資料",
     "trash_dir": "_捨て",
     "moves": [
       {"from": "/Users/you/Downloads/第3回_線形代数.pdf", "to": "線形代数/第3回.pdf", "reason": "正本"},
       {"from": "/Users/you/Desktop/線形代数_第3回.pdf",   "to": "_捨て/線形代数_第3回.pdf", "reason": "上と同一内容の重複"}
     ]
   }
   ```
4. **suggest → preview → 確認 → apply → verify → 報告 → undo** は その場整理と同じ。
   `suggest --target <宛先ルート>` を忘れないこと。`apply`/`preview`/`verify`/`undo`/`redo` の
   位置引数には**宛先ルート**を渡す。undo は集めたファイルを元の散らばっていた場所へ戻す。

## undo（元に戻す）
最新 manifest を逆再生して、全ファイルを元の場所へ復元し、空になったカテゴリフォルダや
`_捨て/` を片付ける。集約モードでも、各ファイルは元々あった場所（絶対パス）へ戻る。
特定の manifest を指定するなら `--manifest <path>`。

## 注意
- スクリプトはどのカレントディレクトリからでも絶対/相対パスで呼べる。例では `scripts/organize.py`
  と書いているが、実際にはこの SKILL.md と同じ場所の `scripts/organize.py` を使うこと。
- 大量ファイルや巨大ファイルが多い場合、ハッシュ計算を省くなら scan に `--no-hash` を付ける。
- 物理削除は一切行わない。ユーザーが明示的に「`_捨て/` を消して」と言った場合のみ、内容を
  確認したうえで削除する。
